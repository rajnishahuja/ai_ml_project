## 1. Stage 1 config has wrong dataset (regression)

**File:** [configs/stage1_config.yaml:10](configs/stage1_config.yaml#L10)

```yaml
dataset: kenlevine/CUAD
```

**Problem:** The canonical dataset is `theatticusproject/cuad-qa`. ARCHITECTURE.md and CLAUDE.md both assert this, and the data loaders use it. The `kenlevine/CUAD` variant is nested SQuAD JSON and is explicitly warned against ("Do **not** use the raw `theatticusproject/cuad` or `kenlevine/CUAD` variants" — ARCHITECTURE.md Key Datasets section).

**Severity:** High (misleading). No active code reads this key today, so it isn't a runtime bug — but the YAML lies about the canonical source, which will mislead the next person who wires Stage 1 training.

**Fix:** One-line YAML change to `dataset: theatticusproject/cuad-qa`. Owner: whoever wrote stage1 config last (`git blame configs/stage1_config.yaml`).

## 4. Stage 4 has a second, divergent reporting path

**Files:**
- [src/stage4_report_gen/report_builder.py](src/stage4_report_gen/report_builder.py) — the **live** Stage 4 (used by FastAPI services and `scripts/run_pipeline.py`).
- [src/stage4_report_gen/nodes.py](src/stage4_report_gen/nodes.py) — a different report builder that uses `print()` with emojis, hardcodes mock scoring, references the experimental `src/workflow/` DAG.

**Problems:**
- Two builders disagree on what a report looks like (`nodes.py` doesn't use the `recommender.py` lookup table; computes a different overall score; emits a different shape).
- `nodes.py` uses `print()` instead of `logging` — violates the convention in ARCHITECTURE.md ("Use `logging` module, not print statements").
- `nodes.py` is only reachable via `src/workflow/graph.py`, which itself can't be instantiated (see issue #5).

**Recommendation:** Either delete `src/stage4_report_gen/nodes.py` along with `src/workflow/` (the DAG isn't on the request path) or rewrite both to use `report_builder.py` and proper logging.

**Severity:** Medium.

## 5. `src/workflow/` LangGraph DAG is broken on main

**Files:**
- [src/workflow/graph.py](src/workflow/graph.py)
- [src/workflow/state.py](src/workflow/state.py)

**Problem:** `graph.py` imports:
```python
from src.stage1_extract_classify.nodes import node_extract_clauses
from src.stage3_risk_agent.nodes  import node_risk_classifier, node_mistral_router
```

Neither `src/stage1_extract_classify/nodes.py` nor `src/stage3_risk_agent/nodes.py` exists on `main`. The DAG cannot be instantiated as-is — `from src.workflow.graph import build_workflow` will raise `ModuleNotFoundError`.

The live request path (FastAPI services + `scripts/run_pipeline.py`) does **not** use this DAG; it calls `assess_clauses()` and `build_report()` directly. So nothing in production is broken — but the experimental code is.

Additionally, `src/workflow/state.py` imports from `app.schemas.domain`, coupling the workflow package to the API layer. If `src/workflow/` is kept, this should be inverted (API depends on workflow, not vice versa).

**Recommendation:**
- (a) Restore the missing `nodes.py` modules and add a smoke test, **or**
- (b) Retire `src/workflow/` entirely along with `src/stage4_report_gen/nodes.py` and the empty `src/stage4_report/` directory.

**Severity:** Medium. Confuses new contributors who try to use the DAG.

## 7. Hardcoded vector count in API endpoint

**File:** [app/routers/documents.py:14](app/routers/documents.py#L14)

```python
"vectors": 3398,
```

**Problem:** The FAISS index size is hardcoded. If the index is rebuilt with a different train split (e.g., after relabel), the API will lie about its own state.

**Fix:** Read the count from the loaded FAISS index at startup (e.g., `index.ntotal`) and cache it in app state. Severity: Low.
